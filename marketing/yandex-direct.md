# Suflo — рекламная кампания в Яндекс Директ

> Рабочий документ по поисковой кампании suflo.ru в Яндекс Директ.

Продукт: **Suflo** — «невидимый» AI-суфлёр для собеседований, лайвкодинга, встреч и звонков
(не виден при демонстрации экрана). Сайт: **suflo.ru**. Деплой сайта см. `../../DEPLOY_RULES.md`
и раздел «Деплой» ниже.

  - Стратегия: **«Максимум конверсий» с оплатой «За конверсии»** (оплата за факт конверсии,

  - Текущая цель оптимизации: **`Begin checkout`** — Метрика, цель `id 574373341`, JS-событие

  - **Цена конверсии (CPA): 300 ₽.**
  - **Недельный бюджет: 8000 ₽.**
  - Гео: **только Россия**. Авто-применение рекомендаций: выключено. UTM на уровне кампании
    (`utm_source=yandex&utm_medium=cpc&utm_campaign={campaign_id}&utm_content={ad_id}&utm_term={keyword}`).
  - Статус: **активна, прошла модерацию** (НЕ черновик).

### Группы (4 интента), посадочные и объявления
Каждая группа: суженный автотаргетинг, ключи, минус-слова, одно **комбинаторное объявление**
(5 заголовков + 3 текста, авто-комбинации), своя посадочная:

| Группа | ID | Посадочная |
|---|---|---|
| Собеседования | 5765385603 | /use-cases/ai-interview-assistant |
| Live coding | 5765942552 | /use-cases/live-coding-assistant |
| Встречи и созвоны | 5765944396 | /use-cases/meeting-assistant |
| Звонки с клиентами | 5765945570 | /use-cases/sales-call-assistant |

Тексты написаны **moderation-safe**: без «невидим / обмани / маскируется» — Яндекс отклоняет
такую формулировку, хотя это и есть суть продукта («незаметен при шеринге экрана»).

## 2. Что было главным блокером

На 2026-06-26 основной блокер был аккаунтного уровня: у `naures1` Яндекс отключал
«оплату за конверсии» до подтверждения номера телефона.

Статус на 2026-06-27:
- Верификация пройдена владельцем.
- Кампания сохраняется.
- Стратегия `За конверсии` активна.
- Текущая проблема уже не в доступности стратегии, а в том, что цель `purchase` была слишком
  редкой для холодного обучения, из-за чего кампания почти не находила трафик под конверсию.

## 3. Цели Метрики на сайте

### 3.1. Покупка

На странице успеха оплаты вызывается `reachMetrikaGoal('purchase')`.
Файл: `app/success/page.tsx`.

Задеплоено на прод. Проверено: `https://suflo.ru/success/` → 200, в бандле
`app/success/page-*.js` присутствует `reachGoal("purchase")`.

### 3.2. Микроцели для переобучения стратегии (добавлено 2026-06-27)

Чтобы не переводить кампанию на оплату за клики по редкой цели `purchase`, на сайт добавлены
ещё 3 JS-события Метрики:

- `reachGoal('register')` — после успешной регистрации в `app/auth/page.tsx`.
- `reachGoal('begin_checkout')` — при первом заходе авторизованного пользователя на `/checkout`
  в рамках сессии, в `app/checkout/page.tsx`.
- `reachGoal('checkout_submit')` — при нажатии кнопки оплаты в `app/checkout/page.tsx`.

Общий helper вынесен в `lib/metrika.ts`.

Практический вывод:
- `purchase` слишком редкая для холодного обучения Поиска.
- Кампания уже переведена с `purchase` на `begin_checkout`, с CPA `300 ₽`, сохранив модель
  оплаты `За конверсии`.
- `checkout_submit` — запасной более жёсткий сигнал, если `begin_checkout` окажется слишком шумной.

## 5. Накопленные знания (для подбора ключей и таргетинга)

- **Категории автотаргетинга** (вкл/выкл по группам): Целевые, Узкие, Широкие, Сопутствующие,
  Альтернативные. У «Собеседований»: **Сопутствующие = мусор** (резюме/zoom) — выключать;
  **Альтернативные = золото** («chatgpt для собеседования») — включать. У «Встреч» Альтернативные ON.
- **Wordstat:** часть изначальных ключей имели ~0 спроса (напр. «помощь на лайвкодинге» = 0).
  Подтверждено в Wordstat; группу «Собеседования» усилили кластером «ии для собеседования».
- **Воронка (известная утечка):** юзеры регистрируются, но не доходят до оплаты (был случай
  116 кликов → регистрации есть, покупок 0). Это отдельная задача — почему отваливаются на
  оплате (смотреть Метрику webvisor + `app/auth`, `app/checkout`). Оплата сама по себе работает
  (проверено в ЮКассе).

## 6. Открытые задачи / следующий шаг

1. **(Наблюдение)** Дать кампании 2-3 дня на переобучение по `begin_checkout`.
2. **(Агент)** Если показов всё ещё почти нет — поднять CPA по `begin_checkout` с `300 ₽`
   до диапазона `400-500 ₽`, не меняя модель оплаты.

4. **(Опц.)** Если `begin_checkout` окажется слишком шумной целью — протестировать переход
   на `checkout_submit` как более жёсткий сигнал.
5. **(Опц.)** Разобрать утечку воронки регистрация→оплата.
